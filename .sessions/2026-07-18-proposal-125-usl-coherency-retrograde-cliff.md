## What happened
- Idea: a coordinated N-agent fleet does NOT scale monotonically. Amdahl's law already caps speedup at 1/α from a serial/contention fraction α (a horizontal asymptote — more agents never worse), but any pairwise **coherency** cost β>0 (the cost of keeping agents consistent with each other, growing as the number of PAIRS ∝ N(N−1)) makes throughput follow the **Universal Scalability Law** `C(N)=N/(1+α(N−1)+βN(N−1))`, whose denominator's coherency term grows as N² while the numerator grows as N — so C(N) is NON-monotone: it rises, **PEAKS at N*=√((1−α)/β), and then DECLINES** ("retrograde scaling"). In completion-time form `T(N)=(1−α)/N+(α−β)+βN` — a shrinking parallel term, a constant, and a GROWING coherency term βN — the analytic optimum is N*=√((1−α)/β), past which every extra agent makes the fleet SLOWER. Counterintuitive core: the optimal fleet is FINITE, and over-provisioning a coordinated job is not merely wasteful (idle agents) but actively counterproductive (the agents spend more time coordinating than working). Two facts the "more agents = more throughput" instinct misses: (1) it is COHERENCY, not contention — β=0 (Amdahl) is monotone up to 1/α and never falls; the retrograde peak appears the instant β>0; (2) the optimum is set by the α:β ratio (N*=31 here), not by how many agents exist. ## 💡 Session idea
"More agents is not more throughput — a coordinated fleet has an OPTIMAL size N*=√((1−α)/β) and adding agents past it makes it SLOWER" is a reusable fleet-sizing lens for any coordinated multi-agent workload. Whenever agents share state (a repo, a queue, a status file), review/merge each other's work, or gossip to stay consistent, they pay a coherency cost that grows with the number of pairs, so throughput follows the Universal Scalability Law and PEAKS then DECLINES — Amdahl (β=0) only tells you it saturates, the USL tells you it reverses. The transferable corrections: (1) never scale a coordinated job to "as many agents as you have" — estimate α (where the speedup curve bends) and β (where it peaks) and size at N*; past the peak, more agents is negative-return; (2) to go faster beyond N* you must cut β (partition state so agents don't touch shared data, batch/serialize merges, replace all-to-all gossip with a hub), which raises N* and the peak — adding agents cannot; (3) only a genuinely independent-subtask job (β≈0) scales with more agents, and even then Amdahl caps it at 1/α. This is the fleet-coordination analogue of the same "diminishing-then-negative returns from adding units" spine that P109 (correlated-fleet variance floor) shows for estimation — but here the curve REVERSES, it doesn't merely floor. Candidate future fleet heads with the same "coordination cost caps or reverses scaling" spine: the Brooks's-law onboarding-throughput dip (adding people to a late project makes it later, via ramp-up + communication overhead), the critical-section / Little's-Law saturation knee, and a lock-convoy / thundering-herd contention collapse.
