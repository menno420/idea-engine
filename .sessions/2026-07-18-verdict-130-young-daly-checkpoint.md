# VERDICT 130 mirror — the Young–Daly optimal checkpoint interval: a long fleet job that can crash and restart from its last checkpoint faces TWO competing wastes — checkpoint OVERHEAD (~C/τ per unit useful work: each checkpoint costs C, taken every τ of work) and lost RECOMPUTE work on a crash (~τ/(2M) per unit work: a Poisson crash at rate 1/M lands on average halfway through the current interval, discarding the work since the last checkpoint). Their sum C/τ + τ/(2M) is a CONVEX falling+rising pair, so total wasted wall time is U-SHAPED in the checkpoint interval τ with a unique INTERIOR optimum where the two wastes are equal: τ*=√(2·C·M) (Young 1974 first-order), overhead floor √(2C/M) (Daly 2006 exact). The "checkpoint as often as you can afford" instinct is WRONG on three counts: (1) the optimum is INTERIOR — checkpointing more often than τ* walks UP the left arm of the U (the C/τ term diverges as τ→0), so over-checkpointing is not "safer", it is wasteful; (2) the two failure modes are SYMMETRIC in leading order — τ*·k and τ*/k inflate the total overhead by the same factor ½(k+1/k); (3) the optimum scales only as √M — a 4× more reliable machine (M→4M) warrants an interval only 2× longer and HALVES the achievable overhead floor (√(2C/M)→½√(2C/M)) — so the checkpoint/snapshot/commit interval is a U-shaped OPTIMISATION, not a one-sided safety dial; the transferable correction is to set the interval to τ*=√(2·C·MTBF) and spend engineering effort on shrinking the checkpoint cost C (incremental/async checkpoints) rather than on raising the frequency, since the optimum scales only as √MTBF. ## 💡 Session idea
This slice pins the Young–Daly law on the FIRST-ORDER checkpoint optimum — the interval τ*=√(2·C·M) that balances a FIXED per-checkpoint cost C against a MEMORYLESS (Poisson, constant-hazard) failure process — and proves the waste is U-shaped and interior. But the two structural assumptions that make τ* clean are exactly the ones production fleets violate, and each violation MOVES the optimum in a direction the √(2CM) rule cannot see. (1) NON-EXPONENTIAL failures: real jobs on real hardware have INCREASING hazard (thermal drift, memory pressure, correlated node degradation over a long run) or heavy-tailed bursty failures, for which the "crash lands halfway through the interval" expectation τ/2 is wrong and the optimum interval should SHRINK as the run ages — a static τ* over-checkpoints early and under-checkpoints late. (2) STATE-DEPENDENT checkpoint cost: C is not fixed — an incremental/differential checkpoint costs less when little has changed, and an async checkpoint overlaps compute, so the effective C(τ) is itself a function of the interval, bending the U and shifting τ* below the √(2CM) prediction. A genuinely different next FLEET slice pins a SECOND checkpoint-scheduling mechanism as a distinct, seed-pinnable head: on the SAME W=40000 job but with a WEIBULL (increasing-hazard, shape κ>1) failure process instead of exponential, pre-register (G-a) a static-τ*-is-suboptimal gate — the constant-interval Young τ*=√(2CM) schedule wastes strictly MORE than a hazard-adaptive schedule that shortens the interval as the instantaneous failure rate rises (penalty ≥ DELTA_MIN by ≥3σ), inverting "τ*=√(2CM) is optimal" once hazard is non-constant; (G-b) an exponential-recovery control — set κ=1 (memoryless) and the adaptive schedule collapses back onto the static τ* (their overhead gap → 0 within 3σ), isolating INCREASING HAZARD as the cause, not the adaptivity machinery; and (G-c) a closed-form anchor — the adaptive optimum's overhead matches the known variable-interval Daly/Bougeret generalisation for Weibull failures within 3σ, so the operator lesson sharpens from "set τ=√(2C·MTBF)" to "√(2C·MTBF) is the constant-hazard special case — under aging/bursty hardware the optimum interval is DECREASING in run age, so budget checkpoint frequency to the instantaneous hazard, not the lifetime MTBF." A seed-pinnable non-exponential-failure checkpoint-scheduling mechanism distinct from this slice's constant-hazard first-order head and from the outbox's single-optimum frame.
