## What this repo is

`idea-engine` is the fleet's ideation lab (per its README and
`docs/decisions.md` [D-0001]). It generates, captures, harvests, probes, and
grooms ideas for the whole ~20-repo fleet so every idea eventually becomes
evidence-checked and built, explicitly parked, or rejected — never orphaned.
Pipeline position (owner ruling Q-0264, cited in the README): `idea-engine`
(generate/probe, mark sim-ready) → **sim-lab** (reproduced-evidence verdict)
→ **fleet-manager** (final review, routes ORDERs) → lanes build. It builds no
product code itself — `docs/architecture.md` states this directly ("No
product code lives here").

## Structure

- `ideas/<section>/` — one directory per active fleet lane (13 sections at
  this snapshot: `fleet`, `gba-homebrew`, `pokemon-mod-lab`, `product-forge`,
  `substrate-kit`, `superbot`, `superbot-games`, `superbot-idle`,
  `superbot-mineverse`, `superbot-next`, `trading-strategy`, `venture-lab`,
  `websites`), each holding dated idea files
  (`ideas/<section>/<slug>-YYYY-MM-DD.md`) with a forward-only state line
  (`captured|probed|sim-ready|parked|rejected|historical`).
- `control/` — the fleet coordination bus: `inbox.md` (manager-written
  ORDERs, append-only, one writer), `status.md` (this repo's own heartbeat,
  one writer = this seat), `outbox.md` (append-only proposals/ASKs to
  sim-lab / fleet-manager), `claims/` (one file per in-flight claim, deleted
  at session close).
- `docs/` — substrate-kit-generated binding docs (architecture, ownership,
  runtime contracts, helper policy, collaboration model, current-state,
  decisions, question-router) plus `docs/retro/` session retrospectives.
- `scripts/` — repo-specific checkers: `check_sections.py`, `check_ideas.py`,
  `check_harvest.py`, `preflight.py` (the single `CHECKS` list the CI gate
  and every session's wake ritual both run — see `scripts/preflight.py`
  lines 87–107).
- `bootstrap.py` (660 KB) — the vendored substrate-kit engine
  (`kit_version: "1.10.0"` per `substrate.config.json`); `CONSTITUTION.md` —
  the generated working agreement; `.substrate/` — kit state, agent
  personas, reflection buffers, and pre-regen backups of KIT-OWNED files.
- `.sessions/` — 193 per-session log files (newest: `2026-07-13-order-008-
  housekeeping.md`), the session-card convention the CI gate enforces.

## CI setup and health

Two workflows, both KIT-OWNED (regenerated by `bootstrap.py adopt/upgrade`,
hand-edits get overwritten — each carries host-customization comment blocks
that must be re-applied after a regen):

- **`substrate-gate.yml`** — the required check. A `control/**`-only diff
  takes a fast lane (control-status + inbox-append-only checks only, reports
  green without the heavy suite). Everything else runs
  `scripts/preflight.py` (the shared check list) plus a session-card gate:
  a PR that **adds** a session card with `Status: in-progress` is a
  designed, intentional **red hold** until a later commit on the same
  branch flips the badge to `complete` — this is documented in-repo (e.g.
  PR #348's body: *"exit 1 on the DESIGNED born-red hold only … This red is
  the designed hold, not a defect"*).
- **`auto-merge-enabler.yml`** — arms native GitHub auto-merge (squash) on
  every non-draft PR whose branch matches an allow-listed prefix
  (`probe/ seed- slice/ groom/ harvest/ build/ upgrade/ refine/ fix/ batch/
  telemetry/ heartbeat/ docs/ claude/`), refuses to arm if the base branch
  requires zero status-check contexts, re-reads the `do-not-automerge` label
  fresh before arming, and skips arming while the PR's own session card is
  still `in-progress` (arms again on the close-out `synchronize` push).

**Verified health:** sampled the last 30 `substrate-gate` runs via the
GitHub Actions API (2026-07-13T21:41Z–22:44Z window): **22 success / 8
failure**. Cross-checked several of the failing runs against their PRs'

## Inconsistencies / issues found

1. **`docs/architecture.md`, `docs/runtime_contracts.md`, `project.index.json`
   carry unfilled substrate-kit placeholder text under a `Status: binding`
   badge** (see Doc quality above). Low severity — the repo has no code
   layers to document — but worth a pass to either fill the sections with
   idea-engine-specific content or downgrade their status badge so
   "binding" isn't claimed for empty prose.
2. **Pre-existing advisory** (already flagged by the repo's own checker,
   not new): `control/status.md`'s `⚑ needs-owner` block lacks the
   OWNER-ACTION fields (`WHAT/WHERE/HOW/WHY-IT-MATTERS/UNBLOCKS/
   VERIFIED-NEEDED`) its own `control/README.md` format mandates —
   `python3 bootstrap.py check --strict` reports this as a non-exit-
   affecting warning. Several recent PR bodies note this same warning as
   "pre-existing … untouched here," so it has been carried for a while
   without being resolved.
3. **`control/outbox.md` is ~400 KB** (413 very-long lines, past the 256 KB
   read-tool limit the seat itself hit) and growing an estimated ~30 KB/day
   per the seat's own **ASK 004** (open to fleet-manager, unanswered as of
   2026-07-13T22:37Z per `control/status.md`). Not a new finding — the
   repo already flagged it — but confirmed still real and still open at
   this snapshot.
4. **`docs/decisions.md` has exactly one entry** (`D-0001`, adopting the
   substrate-kit workflow, dated 2026-07-10) despite ~360 merged PRs of
   operational history since. This is not necessarily wrong (most durable
   calls appear to route through `docs/current-state.md`'s "Ops facts" and
   `control/status.md`'s heartbeat instead), but it means the decisions
   ledger undersells how much has actually been decided — a reader looking
   for "why does this repo work this way" won't find much there.

## Fleet-wide suggestions

1. **Document the "born-red session-card hold" CI semantics once,
   centrally** (e.g. in substrate-kit's shared docs), not per-repo in PR
   bodies. A naive audit of workflow-run conclusions alone would read
   idea-engine's ~27% `substrate-gate` failure rate as a CI health problem;
   it is in fact a designed, self-healing hold pattern. Every repo running
   this kit convention will produce the same misleading raw statistic for
   any external reviewer who doesn't already know the convention.
2. **Give `control/outbox.md`-style append-only logs a shared
   archival/rotation convention in substrate-kit**, rather than each
   high-cadence repo hitting the same 256 KB read-tool wall independently
   and routing an ASK to fleet-manager to invent one. idea-engine already
   flagged this (ASK 004); any other high-velocity coordinator repo in the
   fleet (sim-lab looks like the closest peer by merge cadence) is likely
   approaching the same wall.
3. **Reconcile `project.index.json` / AgentContextPack usage** — either
   populate it for idea-engine's actual `ideas/<section>` areas if the
   context-pack tooling is meant to be live, or note explicitly that it's
   intentionally unused here, so the single placeholder entry doesn't read
   as an abandoned integration to the next auditor.
4. **For any fleet-wide audit/sweep tooling**: re-verify open-PR state
   immediately before any write action, not just at pass start. This
   repo's own PR list changed twice (#361 → #362 merged → #363 merged)
   during the roughly 20 minutes this audit took to write up, purely from
   elapsed real time on a repo this active — a stale read is not a bug in
   the audit, but any write action needs a fresh re-check right before it
   fires (per global safety rule 4).
