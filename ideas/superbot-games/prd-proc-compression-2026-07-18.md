## The phenomenon (one line)
Pseudo-Random Distribution replaces a flat proc chance p with an escalating one — the chance starts at a constant C and rises by C on every consecutive failure, P(proc on the n-th attempt since the last proc) = min(1, C·n), resetting to C on a proc — so that a designer who naively sets the increment C equal to the nominal rate (C = 0.25 "for 25% crit") ships a mechanic whose EFFECTIVE long-run proc rate is 1/E[N] = **0.450704 — nearly double 0.25** — because the escalating chance forces frequent early procs; the correct move is to SOLVE C for the target rate (C ≈ 0.0847 for 0.25), never to set C = the nominal rate.

## The folk belief
"PRD is just crit chance with the streaks smoothed out. If I want a 25% crit rate I set the PRD constant to 0.25 — that's what the number means — and I get roughly 25% crits, just with fewer unlucky dry spells and fewer lucky runs. The anti-streak system changes the VARIANCE of when procs happen, not the RATE." Designers reach for PRD precisely because it feels like "the same rate, minus the bad luck" — Warcraft III and Dota 2 use it for crit / bash / evasion / maim, and community tools expose the increment constant as the tuning knob, which invites reading C as the rate.

## The game-design thesis (reasoned to its fuller form — Q-0254 duty)
The intuition confuses the per-attempt INCREMENT with the long-run RATE. Under PRD the number of attempts N between two procs is a walk whose per-step hazard climbs: the chance on the n-th attempt since the last proc is min(1, C·n), so

    P(N = n) = [ ∏_{k=1}^{n-1} (1 − min(1, C·k)) ] · min(1, C·n),   E[N] = Σ_{n≥1} n·P(N=n),

and the long-run effective proc rate is exactly **1/E[N]** (renewal-reward: procs per attempt = 1 / mean attempts per proc). Because the chance is FORCED up toward 1 as the miss streak grows, the walk is short — most inter-proc gaps are 1, 2, or 3 attempts — so E[N] is well below the 1/C = 4 a flat-C reading would suggest, and 1/E[N] sits well ABOVE C. Concretely for C = 0.25 the walk terminates at n = 4 (C·4 = 1.0, a guaranteed proc):

    P(N=1)=0.25, P(N=2)=0.75·0.5=0.375, P(N=3)=0.75·0.5·0.75=0.28125, P(N=4)=0.75·0.5·0.25·1=0.09375
    (Σ = 1.0), E[N] = 1·0.25 + 2·0.375 + 3·0.28125 + 4·0.09375 = 2.21875,
    effective rate = 1/E[N] = 1/2.21875 = 0.450704.

Two consequences the folk belief misses. **(1) The increment is not the rate — and the gap is large.** Setting C = the nominal rate 0.25 delivers an effective 0.45, an 80% overshoot; the naive knob nearly doubles the realized proc frequency. To actually hit a target rate r you must invert 1/E[N](C) = r for C — a strictly smaller constant. For r = 0.25 the solved constant is **C ≈ 0.0847** (matching the published Dota 2 PRD constant for 25%), roughly a THIRD of the nominal. **(2) What PRD actually buys is a streak bound, not a rate change.** The feature you get for the escalating chance is that the worst-case dry spell is capped at ⌈1/C⌉−1 consecutive misses (for C=0.25, at most 3 misses in a row, because the 4th attempt is a guaranteed proc), whereas a true-random Bernoulli(0.25) stream produces long dry spells (max miss-streak ~50 over a couple million attempts). So the honest reading for a designer: PRD's constant is a curve parameter, not a rate; solve it for the target rate (analytically via E[N] or by simulation), price the tightened bad-luck-streak bound as the actual feature, and never publish or balance around "C = the crit rate" — that ships a proc that fires far more often than intended.

## The formal model (committed constants — sim-lab must reproduce exactly)
- A PRD stream: a running counter `since` = consecutive misses since the last proc (0 at a proc). On each attempt the current attempt index is since+1, so its proc chance is min(1, C·(since+1)); draw U~Uniform[0,1), proc iff U < chance (or chance ≥ 1, a guaranteed proc); on a proc reset since=0, else since+=1.
- **Effective proc rate** = procs / attempts over a stream of ATTEMPTS = 2,000,000 attempts (renewal-reward: the long-run 1/E[N]).
- **Naive regime:** C_NAIVE = 0.25 — the designer sets the increment equal to the nominal rate.
- **Solved regime:** C_SOLVED = the increment whose ANALYTIC effective rate 1/E[N](C) equals the target 0.25, found by bisection on the exact E[N] closed form (E[N] strictly decreasing in C ⇒ 1/E[N] strictly increasing) — the "solve C for the target" fix.
- **True-random control:** a Bernoulli(TARGET=0.25) stream — same nominal rate, no anti-streak — measured for its effective rate (≈0.25) and its max consecutive-miss streak, to contrast streak behaviour.

- Headline significance uses the binomial-proportion se: z = (p_obs − p_null)/√(p_null·(1−p_null)/N) over N = ATTEMPTS (the standard one-sample proportion z the task pins).

## Why it matters (game design)
Anti-streak proc systems are everywhere in competitive games — PRD crit / bash / evasion / maim in Warcraft III and Dota 2, and the same escalating-chance idea in soft-pity gacha ramps and momentum/comeback mechanics. This proposal shows that the tuning knob of such a system — the per-attempt increment C — is NOT the proc rate: the escalating chance forces frequent early procs, so the effective long-run rate is 1/E[N], which for C=0.25 is 0.450704, an 80% overshoot of the nominal. A designer who sets C = the intended rate ships a proc that fires nearly twice as often as intended — a balance error that scales with the whole system (damage, sustain, win-rate). The fix is exact and cheap: solve 1/E[N](C) = target for C (analytically or by simulation) — for a 25% target, C ≈ 0.0847, a third of the nominal — and treat the tightened bad-luck-streak bound (⌈1/C⌉−1 misses, here at most 3 vs true-random's ~50) as the actual feature you are buying, priced separately from the rate. Decision rule for a designer/operator: never set a state-dependent proc's increment equal to its nominal rate; SOLVE the increment for the target long-run rate, verify the realized rate by simulation before shipping, and communicate the streak bound as the feature — a smoother, on-target proc, not a hidden rate inflation.

## Model basis (declared model-dependence — the P024 discipline)
The exact effective-rate identity rests on standard structural choices, all pinned: (a) PRD's canonical form — the proc chance on the n-th attempt since the last proc is exactly min(1, C·n), rising linearly and resetting on a proc (the Warcraft III / Dota 2 definition); (b) independent attempts so N is a proper renewal walk and the long-run rate is exactly 1/E[N] (renewal-reward theorem); (c) the target inversion 1/E[N](C) = r is monotone (E[N] strictly decreasing in C), so C_solved is unique. The QUALITATIVE claim — an escalating-chance proc's effective rate exceeds its increment constant, and setting the increment to the nominal rate overshoots — holds for ANY strictly-increasing per-miss chance curve that resets on a proc (any anti-streak hazard forces early procs, shortening E[N] below 1/C); the linear min(1,C·n) form just makes E[N] a finite exact sum and the seed reproducible. The result depends on the pinned constants: (d) the increment magnitude C sets both the overshoot (larger C ⇒ shorter walk ⇒ larger 1/E[N]/C ratio) and the streak bound ⌈1/C⌉−1; (e) enough attempts (2,000,000) for the binomial se to make the overshoot unambiguous and the max-streak contrast well-resolved. The claim is scoped: under canonical linear PRD with independent attempts, the effective proc rate is 1/E[N] > C, so C = nominal overshoots (here 0.25 → 0.45) and the target-solved constant is C ≈ 0.0847 — demonstrated on the pinned constants, mechanism-explained (renewal-reward on an escalating hazard), not asserted as a universal law.
