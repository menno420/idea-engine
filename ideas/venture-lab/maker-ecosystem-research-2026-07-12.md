## 1. Slicing & presets

- **OrcaSlicer** — [github.com/OrcaSlicer/OrcaSlicer](https://github.com/OrcaSlicer/OrcaSlicer) — Open-source slicer (fork lineage: Bambu Studio → PrusaSlicer → Slic3r) that has become the de-facto community standard for tinkerer printers; profiles are plain **JSON files** with an inheritance system, stored per-user in `user/default/{filament,machine,process}/` and shipped system-wide under `resources/profiles/` — fully text-based and git-versionable ([profile-creation wiki](https://github.com/OrcaSlicer/OrcaSlicer/wiki/how_to_create_profiles)). For him: one slicer that covers both printers, with configs he can commit to his repo and let Claude diff/edit.
- **OrcaSlicer built-in calibration suite** — [OrcaSlicer wiki: Calibration](https://github.com/OrcaSlicer/OrcaSlicer/wiki/Calibration) — Menu-driven calibration tests generated inside the slicer: temperature tower ([temp calib page](https://github.com/OrcaSlicer/OrcaSlicer/wiki/temp_calib)), flow rate, pressure advance ([PA calib page](https://github.com/OrcaSlicer/OrcaSlicer/wiki/pressure_advance_calib), plus [adaptive PA](https://github.com/OrcaSlicer/OrcaSlicer/wiki/adaptive-pressure-advance-calib)), retraction, max volumetric speed; the wiki documents the recommended order (temp → flow → PA → retraction → …). For him: no hunting for calibration STLs — dial in each new filament per printer and save the result straight into a JSON filament profile.
- **OrcaFilamentLibrary (global filament library)** — [resources/profiles/OrcaFilamentLibrary](https://github.com/OrcaSlicer/OrcaSlicer/tree/main/resources/profiles/OrcaFilamentLibrary/filament/SUNLU) — OrcaSlicer's centralized, cross-printer filament library (introduced in [PR #8057](https://github.com/OrcaSlicer/OrcaSlicer/pull/8057)): one filament profile reused across all machines instead of duplicating per printer. For him: define a filament once (e.g. a brand of PLA) and use it on both the small and the 3-color printer.
- **PrusaSlicer** — [github.com/prusa3d/PrusaSlicer](https://github.com/prusa3d/PrusaSlicer) — Prusa's open-source slicer; profiles export as plain-text **.ini config bundles** ([official import/export guide](https://help.prusa3d.com/article/how-to-import-and-export-custom-profiles-in-prusaslicer_382766)), and the official vendor profiles are themselves maintained as a public git repo ([prusa3d/PrusaSlicer-settings](https://github.com/prusa3d/PrusaSlicer-settings)) — proof the format is git-native. For him: an alternative/reference ecosystem; even if he uses Orca, PrusaSlicer's .ini bundles are the cleanest example of slicer-config-in-git.
- **UltiMaker Cura** — [github.com/Ultimaker/Cura](https://github.com/Ultimaker/Cura) — The other big open-source slicer; profiles are text `.inst.cfg` files plus `.xml.fdm_material` material files ([Profiles & Settings wiki](https://github.com/Ultimaker/Cura/wiki/Profiles-&-Settings)), and it has a curated plugin/material [Marketplace](https://marketplace.ultimaker.com/app/cura/plugins). For him: mainly a fallback ecosystem; Orca/Prusa cover his printers better, but Cura's Marketplace has some unique plugins.
- **Teaching Tech calibration site** — [teachingtechyt.github.io/calibration.html](https://teachingtechyt.github.io/calibration.html) — Free browser-based G-code generator for a full calibration ladder (PID tune, e-steps, first layer, flow, temp towers, retraction, linear/pressure advance), generating custom G-code for nearly any printer via JavaScript. For him: the slicer-agnostic complement to Orca's suite — useful for the printer-level (firmware) calibrations Orca doesn't do, like PID and e-steps.
- **Multi-color painting + 3MF color storage** — [3MF Materials Extension spec (3MFConsortium)](https://github.com/3MFConsortium/spec_materials/blob/master/3MF%20Materials%20Extension.md) — 3MF is a ZIP of XML; painted multi-color models store color as per-triangle color-group/material assignments (sRGB `#RRGGBB`), which the slicer maps to physical filament slots — see also [Bambu Lab's doc on standard 3MF color parsing](https://wiki.bambulab.com/en/bambu-studio/Standard-3MF-File-Color-Parsing). OrcaSlicer/Bambu-Studio-family slicers include the paint-on color tool for AMS-style multi-material units. For him: on the 3-color printer, save painted projects as 3MF (never STL) so color assignments, seam paint, and per-plate settings travel with the file — and 3MF project files are just ZIPs, so they can live in his repo.
- **3D Filament Profiles** — [3dfilamentprofiles.com](https://3dfilamentprofiles.com/) — Community site for sharing filament data and print settings plus tracking personal filament inventory; there is even a converter that turns its exports into OrcaSlicer filament profiles ([mhord/OrcaSlicer_Filament_Profile_Generator](https://github.com/mhord/OrcaSlicer_Filament_Profile_Generator)). For him: starting-point settings when he buys a filament brand he hasn't dialed in yet.
- **SliceHub** — [slicehub.org](https://slicehub.org/) — Community-driven library for sharing/downloading slicer profiles across slicers. For him: another source of ready-made profiles; couldn't verify: how large/active its catalog is.
- **Community profile repos on GitHub (pattern)** — e.g. [mjonuschat/OrcaSlicer-Profiles](https://github.com/mjonuschat/OrcaSlicer-Profiles) — Many makers keep their OrcaSlicer printer/filament/process JSONs in personal GitHub repos; a common, working pattern. For him: the exact model to copy — a `profiles/` folder in his repo, exported from Orca, versioned and editable by Claude. (Orca itself also ships vendor profiles in-tree, so upstreaming a fix via PR is possible.)

## 2. Farm/queue tooling

- **OctoPrint** — [octoprint.org](https://octoprint.org/) / plugin registry at [plugins.octoprint.org](https://plugins.octoprint.org/) — The classic open-source web interface for G-code printers (one instance per printer, typically on a Raspberry Pi), with a huge plugin ecosystem. For him: the lowest-friction way to put both printers on the network with a REST API that scripts (or Claude-written tools) can drive.
- **Notable OctoPrint plugins for a 2-printer setup** —
  - **Bed Level Visualizer** — [github.com/jneilliii/OctoPrint-BedLevelVisualizer](https://github.com/jneilliii/OctoPrint-BedLevelVisualizer/) ([registry entry](https://plugins.octoprint.org/plugins/bedlevelvisualizer/)) — renders the firmware's bed-mesh as a 3D surface; instant answer to "is my bed the problem?".
  - **PrintTimeGenius** — [github.com/eyal0/OctoPrint-PrintTimeGenius](https://github.com/eyal0/OctoPrint-PrintTimeGenius) ([registry entry](https://plugins.octoprint.org/plugins/PrintTimeGenius/)) — G-code pre-analysis for print-time estimates often within ~0.2%; useful when juggling jobs across two machines.
  - **Continuous Print** — [github.com/smartin015/continuousprint](https://github.com/smartin015/continuousprint) ([registry entry](https://plugins.octoprint.org/plugins/continuousprint/)) — per-printer print queue with auto bed-clearing scripts and job grouping. Note: last release v2.4.1 (Apr 2024) with many open issues — works but maintenance looks slow.
- **Klipper** — [klipper3d.org discourse](https://klipper.discourse.group/) / firmware widely distributed, e.g. via the [klipper GitHub topic](https://github.com/topics/klipper) — Replacement firmware that runs the motion planning on a Pi/SBC and turns the printer's own board into a dumb stepper driver; unlocks input shaping, pressure advance, macros. Consumer printers with common boards (Ender-3 family incl. Creality 4.2.2 and BTT SKR Mini E3, and most open-hardware machines) can be flashed — see e.g. this [Ender 3 V2 Klipper guide](https://github.com/BenD780x9/Klipper-for-Ender_3_v2). For him: if either of his printers is a Creality-style open machine, Klipper is the biggest single upgrade available; closed ecosystems (e.g. Bambu) can't be flashed.
- **Moonraker** — [github.com/Arksine/moonraker](https://github.com/Arksine/moonraker) — Python web API server that exposes Klipper over HTTP/WebSocket; the integration point for Mainsail/Fluidd, Spoolman, timelapse, and farm tools. For him: this API is what makes a Klipper printer scriptable from his repo.
- **Mainsail & Fluidd** — [github.com/mainsail-crew/mainsail](https://github.com/mainsail-crew/mainsail) and [github.com/fluidd-core/fluidd](https://github.com/fluidd-core/fluidd) — The two main web UIs on top of Moonraker/Klipper (pick one; both actively maintained). Install of the whole Klipper/Moonraker/UI stack — including **multiple Klipper instances on one Pi for multiple printers** — is automated by **KIAUH** ([github.com/dw-0/kiauh](https://github.com/dw-0/kiauh), [docs](https://dw-0.github.io/kiauh/)). For him: one Raspberry Pi could run both printers as two Klipper instances via KIAUH.
- **FDM Monster (multi-printer dashboard; OctoFarm's successor)** — [github.com/fdm-monster/fdm-monster](https://github.com/fdm-monster/fdm-monster) — Farm-management platform connecting OctoPrint, Moonraker, PrusaLink and Bambu Lab printers in one grid view; explicitly announced as the FOSS replacement for OctoFarm ([announcement issue](https://github.com/fdm-monster/fdm-monster/issues/1678)). The older **OctoFarm** ([github.com/OctoFarm/OctoFarm](https://github.com/OctoFarm/OctoFarm)) is abandoned (development suspended) — avoid. For him: the single-pane-of-glass for his two printers if he wants one dashboard instead of two browser tabs.
- **SimplyPrint** — [simplyprint.io](https://simplyprint.io/) ([print-queue feature](https://simplyprint.io/features/print-queue)) — Cloud farm-management service with a cross-printer queue that auto-matches jobs to a suitable free printer; integrates with OctoPrint, Klipper/Moonraker/Mainsail/Fluidd, Bambu and Prusa. Free tier exists; it's a hosted service, not self-hosted. For him: the easiest true "one queue, two printers" experience if he's OK with cloud.
- **Spoolman** — [github.com/Donkie/Spoolman](https://github.com/Donkie/Spoolman) — Self-hosted filament-spool inventory database with web UI, REST API, and QR-code spool labels; auto-decrements spool weight as prints run via Klipper/Moonraker or the OctoPrint plugin ([mdziekon/octoprint-spoolman](https://github.com/mdziekon/octoprint-spoolman); [usage-tracking wiki](https://github.com/Donkie/Spoolman/wiki/Automatic-Filament-Usage-Tracking)). The de-facto standard; older per-instance alternatives (OctoPrint FilamentManager) are effectively superseded. For him: one shared filament database serving both printers — and a clean REST API Claude can query ("how much of the red PETG is left?").
- **Obico (formerly The Spaghetti Detective)** — [github.com/TheSpaghettiDetective/obico-server](https://github.com/TheSpaghettiDetective/obico-server) / [obico.io](https://www.obico.io/) — Open-source AI print-failure detection (webcam frames scored by an ML model; alerts/auto-pause on spaghetti) plus remote monitoring; integrates with OctoPrint and Klipper; can be fully self-hosted (the server repo) or used as a cloud service. For him: watchdog for long multi-color prints, which fail expensively.
- **OctoEverywhere + Gadget AI** — [octoeverywhere.com](https://octoeverywhere.com/) ([OctoPrint plugin](https://plugins.octoprint.org/plugins/octoeverywhere/), [Klipper support](https://octoeverywhere.com/klipper)) — Free unlimited remote access to OctoPrint/Klipper/Bambu/Elegoo printers plus "Gadget," its free AI failure-detection assistant. Cloud service (closed backend), very low-effort setup. For him: the zero-maintenance alternative to self-hosting Obico.
- **Timelapse tools** —
  - **Octolapse** — [github.com/FormerLurker/Octolapse](https://github.com/FormerLurker/Octolapse) — stabilized timelapses for OctoPrint (moves the head to a fixed position each layer); still maintained (v0.4.5 Nov 2024, activity through 2025).
  - **moonraker-timelapse** — [github.com/mainsail-crew/moonraker-timelapse](https://github.com/mainsail-crew/moonraker-timelapse) — the Klipper-world equivalent, a Moonraker component with layer-macro and hyperlapse modes, rendered via ffmpeg (pre-installed on MainsailOS/FluiddPi).
  - For him: pick whichever matches the printer's stack; both are the standard choice on their side.

## 3. Model discovery & parametric design

### Model platforms (API / feed reality check)

- **Thingiverse API** — [thingiverse.com/developers](https://www.thingiverse.com/developers) — Official, documented public REST API (OAuth token per account; [getting-started guide](https://www.thingiverse.com/developers/getting-started), [Swagger reference](https://www.thingiverse.com/developers/swagger)) with read and write endpoints for things, files, search; community wrappers exist, e.g. [python-thingiverse](https://github.com/garciajg/python-thingiverse). For him: the one platform where a Claude-written Python script can legitimately search/download models today.
- **Printables** — [printables.com](https://www.printables.com) — No official public API or RSS feed; the site runs on a GraphQL backend that community projects tap unofficially, e.g. [PrintablesGraphQL](https://github.com/100prznt/PrintablesGraphQL), and third parties publish JSON-Feed style feeds ([trending](https://dyn.tedder.me/rss/printables/trending.json), [top rated](https://dyn.tedder.me/rss/printables/top_rated.json) — documented in [this writeup](https://jmtd.net/log/printables_feed/)). For him: usable for a "what's trending" digest script, but it's unofficial and could break; couldn't verify: any official Prusa-sanctioned API.
- **MakerWorld (Bambu Lab)** — [makerworld.com](https://makerworld.com) — No documented public API; users asking on the [official Bambu forum](https://forum.bambulab.com/t/public-api-for-makerworld/52699) found none, and only paid third-party scrapers exist (e.g. [Apify MakerWorld scraper](https://apify.com/stealth_mode/makerworld-models-details-scraper/api/python)). For him: relevant because his 3-color printer is likely Bambu-ecosystem; treat MakerWorld as browse-only, not scriptable. Couldn't verify: any RSS feed.
- **Thangs** — [thangs.com/explore](https://thangs.com/explore) — Search engine that indexes models across platforms (including geometric "search by shape"). For him: good manual discovery front-end; couldn't verify: a public developer API or RSS feed for Thangs.

### Parametric design he can drive with Claude

- **OpenSCAD** — [openscad.org](https://openscad.org) — Declarative CSG scripting language (its own `.scad` language, not Python); files are pure text and diff beautifully; headless render/export via `openscad -o out.stl file.scad`; low learning curve for boxy functional parts, gets awkward for fillets/organic shapes. For him: the lingua franca of maker parametric design — the largest ecosystem of existing generators (Gridfinity, box makers, BOSL2) is written in it, so Claude can modify battle-tested community code rather than starting from scratch.
- **CadQuery** — [github.com/CadQuery/cadquery](https://github.com/CadQuery/cadquery) — Python library on the OCCT B-rep kernel; models are plain `.py` files (fully git-friendly); exports STL, STEP, DXF, AMF, 3MF from a script; moderate learning curve (fluent "workplane" API). For him: a real CAD kernel means real fillets, chamfers and STEP export that OpenSCAD's mesh-based CSG can't do; actively maintained.
- **build123d** — [github.com/gumyr/build123d](https://github.com/gumyr/build123d) — Younger, more Pythonic redesign derived from CadQuery ideas, same OCCT kernel ([docs](https://build123d.readthedocs.io/)); plain Python source, `export_stl(part, "part.stl")` one-liner; actively developed ([PyPI](https://pypi.org/project/build123d/)). For him: arguably the cleanest code-CAD API for an LLM to write, though its ecosystem/examples corpus is smaller than OpenSCAD's or CadQuery's.
- **Fusion 360 scripting** — [Autodesk Fusion API docs](https://help.autodesk.com/view/fusion360/ENU/?guid=GUID-A92A4B10-3781-4925-94C6-47DA85A4F65A) — Python (or C++) add-in API driving the full Fusion GUI app ([API overview](https://aps.autodesk.com/developer/overview/autodesk-fusion-api)); scripts are text but must run *inside* a running Fusion desktop instance — no headless render, and the `.f3d` design files themselves are binary. For him: fine if he already lives in Fusion, but the worst fit for a repo + Claude loop: no CI rendering, binary artifacts, steepest API learning curve.

**Verdict — most Claude-friendly:** OpenSCAD for leverage, build123d/CadQuery for ambition. OpenSCAD wins on ecosystem: nearly every reusable generator he'd want (Gridfinity, box makers, threads) is already `.scad`, renders headlessly in CI with one command, and Claude has seen enormous amounts of it. For parts needing fillets/STEP or where he prefers real Python, build123d is the nicest authoring experience and equally git/CI-native. Fusion 360 scripting is the only clearly poor fit (GUI-bound, binary files).

### Starter generators (real repos)

- **Ultimate Box Maker** — [github.com/jbebel/Ultimate-Box-Maker](https://github.com/jbebel/Ultimate-Box-Maker) — Classic customizable OpenSCAD electronics-enclosure generator (revised fork of Heartman's Thingiverse original; an extended [Advanced Ultimate Box Maker](https://github.com/M-oster/Advanced-Ultimate-Box-Maker) builds on BOSL2/NopSCADlib). For him: instant project boxes for his Arduino builds — set dimensions/vents/PCB standoffs as parameters, render, print.
- **gridfinity-rebuilt-openscad** — [github.com/kennetek/gridfinity-rebuilt-openscad](https://github.com/kennetek/gridfinity-rebuilt-openscad) — Ground-up OpenSCAD rebuild of Zack Freedman's Gridfinity bin system ([docs site](https://kennetek.github.io/gridfinity-rebuilt-openscad/)); a feature-richer sibling is [gridfinity_extended_openscad](https://github.com/ostat/gridfinity_extended_openscad). For him: workshop organization on demand — Claude can generate custom-divided bins sized to his exact parts drawers.
- **BOSL2** — [github.com/BelfrySCAD/BOSL2](https://github.com/BelfrySCAD/BOSL2) — The Belfry OpenSCAD Library v2: shapes, attachments, gears, and printable threads via [threading.scad](https://github.com/BelfrySCAD/BOSL2/wiki/threading.scad) and [screws.scad](https://github.com/BelfrySCAD/BOSL2/blob/master/screws.scad) (UTS/ISO, ACME, pipe, buttress threads); actively maintained. For him: the single most useful library to vendor into his repo — threaded lids, knobs, and screw-together parts become a few lines of code.
- **NopSCADlib** — [github.com/nophead/NopSCADlib](https://github.com/nophead/NopSCADlib) — Large OpenSCAD library of vitamins (fasteners, electronics components, extrusions) and printed-part helpers, aimed at documenting whole machines. For him: useful when a design must fit real hardware (PCBs, screws, bearings) he already owns.
- **OpenSCAD library index** — [openscad.org/libraries.html](https://openscad.org/libraries.html) — Official curated list of OpenSCAD libraries. For him: one bookmark that leads to gears, boxes, joints and more when a project needs a part family not covered above.

## 4. Arduino & robot arm

- **PlatformIO** — [docs.platformio.org: Arduino vs PlatformIO](https://docs.platformio.org/en/latest/faq/arduino-vs-platformio.html) — Cross-platform embedded build system: a repo is a `platformio.ini` + `src/` + `lib/`, with per-environment board configs, pinned library versions, unit testing, and first-class CI support. For him: the most git-native way to structure his Arduino projects — `pio run` builds everything reproducibly, dependencies are declared in text, and Claude can edit `platformio.ini` like any config file.
- **Arduino CLI** — [github.com/arduino/arduino-cli](https://github.com/arduino/arduino-cli) — Official command-line Arduino toolchain (docs: [sketch build process](https://docs.arduino.cc/arduino-cli/sketch-build-process)); repo-based CI is typically the official [arduino/compile-sketches GitHub Action](https://github.com/arduino/compile-sketches), which compiles every sketch in the repo and reports memory deltas ([Arduino blog announcement](https://blog.arduino.cc/2021/04/09/test-your-arduino-projects-with-github-actions/)). For him: lighter-weight and IDE-compatible; fine if he wants to stay close to stock Arduino, but library pinning and multi-board setups are weaker than PlatformIO's. Verdict: PlatformIO for the repo he'll share with Claude; arduino-cli/compile-sketches as the minimal alternative.
- **Wokwi + Wokwi CI** — [docs.wokwi.com: Wokwi CI getting started](https://docs.wokwi.com/wokwi-ci/getting-started) — Browser-based Arduino/ESP32 simulator that also runs headlessly: a `wokwi.toml` + `diagram.json` in the repo, run locally via wokwi-cli or in CI via the official [wokwi/wokwi-ci-action](https://github.com/wokwi/wokwi-ci-action) ([GitHub Actions guide](https://docs.wokwi.com/wokwi-ci/github-actions)), including YAML automation scenarios that press virtual buttons and assert on serial output. For him: the killer piece for a Claude workflow — Claude can write firmware, *simulate it on a virtual Uno in CI*, and check serial output, all without the physical board plugged in. (Cloud simulation needs a Wokwi CI token; free tier exists.)
- **Robot-arm serial protocols & G-code-style control** — [github.com/cgxeiji/CGx-Gcode-RobotArm](https://github.com/cgxeiji/CGx-Gcode-RobotArm) — Arduino G-code decoder for robot arms (G0/G1 moves to XYZ) that pairs with the same author's [CGx-InverseK](https://github.com/cgxeiji/CGx-InverseK) 3-link+base analytic IK library (tested on Uno and the Arduino Braccio); similar projects: [ThisArm_Firmware](https://github.com/JoLamMaker/ThisArm_Firmware) (G-code firmware for hobby arms) and [robotarm-6dof-rcservo-atmega2560](https://github.com/steinkohl/robotarm-6dof-rcservo-atmega2560) (6-DOF RC-servo arm with rudimentary G-code + ROS messages over USB). For him: the common hobby pattern is exactly this — plain-text G-code-ish commands over USB serial (115200 baud typical), ideal because Claude can generate motion scripts as reviewable text files. Note: these are hobby repos with modest/idle maintenance; treat as reference implementations to fork rather than supported products.
- **Python-side inverse kinematics: IKPy** — [github.com/Phylliade/ikpy](https://github.com/Phylliade/ikpy) — Pure-Python universal IK/FK library (numpy/scipy, chains definable by hand or from URDF, Python 3 only). For him: the standard "run IK on the laptop, send joint angles to the Arduino over serial" stack for 4-6 DOF hobby arms — see [smart-methods/robot_arm_arduino_python](https://github.com/smart-methods/robot_arm_arduino_python) for exactly this Arduino+ikpy pattern, and [easyEEZYbotARM](https://github.com/meisben/easyEEZYbotARM) for a full Python+Arduino kinematics stack for the 3D-printed EEZYbotARM (a printable arm he could make on his own printers). Lighter alternative: [tinyik](https://github.com/lanius/tinyik) (tiny scipy-based IK solver).
- **pySerial** — [github.com/pyserial/pyserial](https://github.com/pyserial/pyserial) — The de-facto Python serial-port library. Maintenance state: last release is [v3.5 (Nov 2020)](https://github.com/pyserial/pyserial/releases/tag/v3.5); repo shows occasional recent commits but release cadence is effectively dormant — still ubiquitous, stable, and safe to depend on. For him: backbone of any laptop→Arduino link — Claude writes a 20-line pyserial script and the arm moves.
- **Firmata / pyFirmata2** — [github.com/berndporr/pyFirmata2](https://github.com/berndporr/pyFirmata2) — Firmata is a standard protocol for driving Arduino pins live from a host; the original [pyFirmata](https://pypi.org/project/pyFirmata/) stalled (last release 2019), while pyFirmata2 adds callback-based sampling and had a release in late 2024 ([PyPI](https://pypi.org/project/pyFirmata2/)) though Snyk rates its activity [inactive-ish](https://snyk.io/advisor/python/pyfirmata2). For him: good for quick "Python directly toggles pins / reads sensors, no firmware edits" experiments; for anything serious, prefer custom firmware + pyserial.
- **Core tinkerer libraries** — [ArduinoJson](https://github.com/bblanchon/ArduinoJson) (JSON for embedded C++, MIT, very actively maintained — makes structured serial protocols between Claude-written Python and the Arduino trivial); [FastLED](https://github.com/FastLED/FastLED) (the standard addressable-LED animation library); [AccelStepper](https://www.airspayce.com/mikem/arduino/AccelStepper/) (acceleration-controlled stepper driving — the default answer for smooth multi-motor motion); [Adafruit Unified Sensor](https://www.arduinolibraries.info/libraries/adafruit-unified-sensor) (common interface underpinning most Adafruit sensor drivers). For him: these four cover the bulk of "sensor + motion + lights + talk to the laptop" tinkering.

### Control stack for the generic 6DOF/6-servo arm kit

His arm is a generic no-brand 6DOF kit (6 servos, MG996R-class, 180° nominal
range, gripper claw, **no controller board included** — Amazon.nl class).
Branded ecosystems (uArm/Dobot/xArm/LewanSoul bus-servo firmware) are
deliberately excluded — none of it applies to this hardware.

#### Servo driver board — PCA9685 16-channel + external power

- **Adafruit PCA9685 guide — "Using the Adafruit Library"** — [learn.adafruit.com](https://learn.adafruit.com/16-channel-pwm-servo-driver/using-the-adafruit-library) — Official walkthrough for the `Adafruit_PWMServoDriver` Arduino library (I2C, 2 pins, up to 16 servos, examples ship in the IDE library manager). For him: the canonical starting sketch for driving all 6 arm servos from one board.
- **Adafruit PCA9685 guide — "Hooking it Up"** — [learn.adafruit.com](https://learn.adafruit.com/16-channel-pwm-servo-driver/hooking-it-up) — Wiring page; states explicitly that powering servos from the Arduino 5V pin causes "electrical noise and brownouts from excess current draw" that can reset or overheat the Arduino, and that the V+ servo rail must be fed separately from chip VCC. For him: the single most important wiring fact for this build.
- **Arduino Forum: powering PCA9685 with MG996R + SG90 mix** — [forum thread](https://forum.arduino.cc/t/how-to-power-pca9685-16-channel-12-bit-servo-driver-to-control-3-mg996r-servo-motors-and-3-sg90-micro-servo-motors/1247160) — Community thread sizing an external supply for exactly this servo mix; rule of thumb from the thread ecosystem: ~5V–6V at high amperage (a 5V 10A class supply for six servos), MG996R can pull ~1.4A+ each under load. - **Arduino Forum: "Robotic Arm servos not working PCA9685"** — [forum thread](https://forum.arduino.cc/t/robotic-arm-servos-not-working-pca9685/1232503) — Troubleshooting thread for the classic failure mode (servos dead or twitching on PCA9685); recurring fixes are common ground between external supply and Arduino, and adequate supply current. For him: a pre-made debugging checklist for day one.
- **DroneBot Workshop forum: PCA9685 + external power not working** — [forum thread](https://forum.dronebotworkshop.com/motors/pca9685-servo-motors-not-working-with-external-power/) — Another common-ground / V+ rail troubleshooting thread; DroneBot Workshop's servo material is a credible hobbyist reference. For him: useful second source when wiring "should work" but doesn't.
- **Seeed blog: 16-channel PCA9685 controlling a robotic arm** — [seeedstudio.com blog](https://www.seeedstudio.com/blog/2019/09/05/get-started-with-the-grove-16-channel-pwm-driver-pca9685-and-control-your-own-robotic-arm/) — End-to-end example of a multi-servo arm on a PCA9685 (Grove variant, same chip). For him: shows the full arm-on-one-driver pattern he'll replicate.

**Why external power matters (summary, all linked above/below):** an Uno's onboard regulator supplies ~500 mA total, while a single MG996R-class servo stalls at ~2.5 A ([servodatabase](https://servodatabase.com/servo/towerpro/mg996r), [handsontec datasheet](https://www.handsontec.com/dataspecs/motor_fan/MG996R.pdf)); six of them moving together will brown out any USB/regulator supply instantly. PCA9685's separate V+ screw-terminal rail exists exactly for this: chip logic from Arduino 5V, servo power from a dedicated 5–6 V high-current supply, grounds tied together.

7. **PCA9685 + external 5–6 V supply + ServoEasing + per-joint calibration** ([Adafruit guide](https://learn.adafruit.com/16-channel-pwm-servo-driver/using-the-adafruit-library), [ServoEasing](https://github.com/ArminJo/ServoEasing), [calibration procedure](https://zbotic.in/servo-motor-calibration-setting-min-max-center-position/)) — his arm kit ships with no controller, so this is the hardware decision everything else sits on: one driver board for all six servos, a dedicated supply that prevents the 6× MG996R brownout by design, smooth easing that also flattens current spikes, and measured per-joint pulse limits that keep clone gears alive.
8. **pySerial + IKPy arm stack over a simple text serial protocol** ([pyserial](https://github.com/pyserial/pyserial), [ikpy](https://github.com/Phylliade/ikpy), [protocol pattern](https://microbotlabs.com/create-and-code-a-simple-serial-command-protocol-for-your-arduino-robot.html)) — laptop-side Python does the kinematics from ruler-measured link lengths, the Arduino just executes joint angles over a line-based G-code-ish protocol; every motion script becomes a reviewable text file Claude can write, log, and replay.
9. **Obico self-hosted** ([repo](https://github.com/TheSpaghettiDetective/obico-server)) — open-source AI failure detection to babysit long multi-color prints, which fail expensively; OctoEverywhere's Gadget is the zero-effort cloud fallback, and FDM Monster adds the two-printer dashboard on the same camera investment (Spoolman is the same-family inventory pick once the stack is running).
10. **The arm-clears-the-bed gap** ([closest prior art](https://hackaday.io/project/181649-a-robot-for-automating-3d-printers)) — verified to be mostly unsolved in hobby open source; his exact hardware combo (6DOF arm + small printer) makes it a credible, novel flagship project rather than a re-tread — the crossover that ties the whole gift together.
