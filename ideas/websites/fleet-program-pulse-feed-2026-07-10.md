## Problem

PROPOSAL 002's probe routes the story page (phase 1) with "2–3 real git-history
graphs … generated by a build-time script into a committed data file" (probe Q8) —
scoped to superbot's history. But the same probe says the story page is "the public
face of the whole EAP experiment" (Q5) and flags, unclaimed, that "the
git-history-graphs build step would work for any fleet repo" (Q2). The experiment's
real story is ~15 repos, 10 lanes, hundreds of merged PRs in days — and no artifact
anywhere produces *historical* program-wide metrics: the control-plane's `/fleet`
renders live heartbeats only, now with no trend.

## Idea

Build phase 1's graph exporter fleet-general from its first commit: one stdlib
script that walks any repo's git history (blobless clone — the proven cheap
transport) and emits one schema-stamped `pulse.json` per repo — PRs merged over
time, commits/day, test-count trend, session-card count, first/last activity —
plus a fleet rollup (lanes born over time, total merge rate). Committed, versioned,
fail-closed on schema mismatch: exactly the `dashboard.json` committed-feed shape
the lane already consumes in `dashboard/data_source.py` ("reads only the committed
generated artifacts … fetched live over raw.githubusercontent.com — never fake
data"). The story page renders the fleet-wide graphs from it; `/fleet` can add
trend sparklines later off the same feed for free; and the story's closing chapter
can render the lane's existing live `/activity` feed — the story literally still
being written (that fold-in replaces a separate "living epilogue" capture).

**Why now:** phases 1–2 are routable to the lane now, without the sim verdict
(probe recommendation) — the exporter's shape gets decided at the story page's
FIRST commit, and a one-off superbot-only script is the cheap-today choice that
costs a rewrite the moment the fleet story (or any second consumer) is wanted.
